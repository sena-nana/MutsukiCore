# v0.2 Phase C 阶段记录 —— 多 Agent 广播与配置校验

## 阶段目标

补齐 v0.2 通用 Agent 框架的 Phase C：让多个 Agent 可以在同一进程内通过 `AgentRegistry` 接收同一 envelope 广播，并把插件配置从“调用方必须传 Struct”推进到“loader 可接收 raw mapping 并做 schema 校验”。

## 完成项

- `mutsukicore/core/agent_registry.py`：新增进程内 `AgentRegistry`，以弱引用保存 Agent，避免全局注册表造成测试 / 生命周期泄漏。
- `Agent.__post_init__`：Agent 构造后自动登记到 `AgentRegistry`。
- `Dispatcher.publish()`：由单 Agent 投递改为枚举所有 `phase == AWAKE` 且 `accepts` 命中的 Agent，并逐个投递到 inbox。
- `mutsukicore/plugins/cross_agent_smoke.py`：新增跨 Agent 冒烟入口，验证 control / audit 两个 Agent 同时收到一条 IM envelope。
- `PluginLoader.load_into(..., configs=Mapping[str, object])`：支持 raw mapping / payload，通过 `msgspec.convert(..., type=cls.Config)` 转换为目标 Config。
- `Errs.PLUGIN_CONFIG_INVALID`：配置转换失败时 fail-loud，并由 `PluginLoadFailedError` 统一包装。

## 关键决策

- `AgentRegistry` 使用弱引用，而不是强持有 Agent；测试中仍可 `AgentRegistry.clear()` 做隔离。
- `Dispatcher.publish()` 广播目标仅限 awake Agent；`STOP` / `SLEEP` 阶段不会继续接收 envelope。
- 配置错误使用 `plugin.config_invalid`，但外层保持 `PluginLoadFailedError`，调用方不用为配置错误走另一条异常处理路径。
- loader 的 `configs` 接口刻意接受 `Mapping[str, object]`，为后续 YAML / TOML 配置文件管线留下无迁移成本的入口。

## 验证

阶段收尾时应运行：
